How to Develop: 101

A simple, reliable process for choosing an idea, shipping a prototype, and staying sane while you do it.

1) Pick a project idea

Start with a real pain. The best hackathon ideas are small problems you understand personally: school logistics, clubs, sports teams, studying, transportation, food, schedules.

Make it a story: “Before” (frustration) → “After” (your app makes it easy). Judges remember stories.

Aim for a demoable change. If you can’t show the problem being solved in 30 seconds, the idea is probably too abstract.

Quick filters

  • Demo: Can we show it working live?
  • Scope: Can 2–4 people build a prototype?
  • Data: Do we have sample data?
  • Impact: Who benefits, and how?

2) Scope for a hackathon timeline

Don’t plan the “final app”. Plan the smallest prototype that proves the core value.

MVP

Must have

The one feature your demo depends on.

NICE

Nice to have

Polish if time remains (UI, extras).

NOPE

Not now

Anything that risks the MVP.

3) Basic Git workflow (safe + simple)

One person sets up the repo (GitHub/GitLab) and invites everyone.

Work in small pieces. Don’t make a 3-hour change without pushing. Push often.

Pull before you start. It avoids messy conflicts.

git checkout -b feature/my-change
git add .
git commit -m "Add team form"
git push -u origin feature/my-change

If you’re not using PRs, agree on one rule: only one person edits the same file at the same time.

Team split ideas

  • Builder 1: UI + pages
  • Builder 2: Data + APIs
  • Builder 3: Demo flow + pitch
  • Builder 4: Testing + polish

4) Split work across the team

The fastest teams agree on a plan, then work in parallel without stepping on each other.

Write the demo script early

If you can narrate the demo, you know what you need to build. Let the pitch guide the product.

Define “done” in one sentence

Example: “User can create a team, choose mentors, and see a confirmation.”

Keep a tiny task board

To Do → Doing → Done. The board is the truth, not memory.

Avoid “merge hell”

Split by files/pages, push frequently, and pull before starting new work.
